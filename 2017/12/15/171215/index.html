iOS 中 32位与64位 数据类型的问题

2017-12-15

32位与64位 的数据类型对比

本次遇到的问题

  在开发中，用到一个 NSInteger 的变量，但是长度在某些情况下超过了 NSInteger 的范围，造成了部分数据的访问不正常。
   iOS 中对 NSInteger 的定义

  在苹果的 api 实现中，NSInteger 是一个封装，它会识别当前操作系统的位数，自动返回最大的类型。
  NSInteger 在32位时等同于 int，在64位时等同于 long
即：
   32 位上面 long 和 int 都占 4 字节。 NSInteger 表示 int -2147483648 ~ 2147483647
   64 位上面 long 8字节，int 4 字节。NSInteger 表示 long -9223372036854775808 ~ 9223372036854775807

  该现象会在 5s 之前32位机型中会出现，在安卓32位机器中出现，在调试几次后发现是因为数据类型的位数不够，造成了数据访问不正常。后修改为 long long 类型后，数据正常。
  在一次后端工程师设计中，计算价格的时候，我们经过测试，输入8位以后的数字，后面的部分会不准确，后经过几次测试发现，是因为后端对此字段的定义数据类型为 float 造成的，后改为 double 类型，数据正确，这些问题都是在设计之初，没有充分考虑到数据的长度会影响结果而造成的。

特此记录

不同位数机器的字节数

可用如sizeof（char),sizeof(char*)等得出

  • 32位编译器
    char ：1个字节
    char*（即指针变量）: 4个字节（32位的寻址空间是2^32, 即32个bit，也就是4个字节。同理64位编译器）
    short int : 2个字节
    int： 4个字节
    unsigned int : 4个字节
    float: 4个字节
    double: 8个字节
    long: 4个字节
    long long: 8个字节
    unsigned long: 4个字节

  • 64位编译器
    char ：1个字节
    char*(即指针变量): 8个字节
    short int : 2个字节
    int： 4个字节
    unsigned int : 4个字节
    float: 4个字节
    double: 8个字节
    long: 8个字节
    long long: 8个字节
    unsigned long: 8个字节
